# Notion Scores Backfill Plan

Goal: Extract four category scores from AI-generated reports in Notion pages and backfill page properties without normalization. Handle multiple reports per page (choose the highest), skip unrelated 1–4 scoring formats, limit processing to entries created after 2025-06-01, and mirror updates to the external Notion database when available.

## Target Properties (Notion)
- Founder Pattern
- Idea Pattern
- Structural Advantage
- Asymmetric Signal

Recommendation: Set these four properties as `number`. Store the explicit numeric score (the leading value in a fraction/percent) while preserving the original raw string in the audit logs. If the properties currently use another type, the script logs and skips them.

## Data Window
- Only process pages created after 2025-06-01.
  - Notion filter: `{ "timestamp": "created_time", "created_time": { "after": "2025-06-01T00:00:00Z" } }`
  - Optionally support a flag to switch to `last_edited_time`.

## Batch Retrieval and Pagination
- Notion supports fetching multiple database entries at a time.
  - Use `databases.query` with `page_size` up to 100 per request.
  - Iterate with `start_cursor` while `has_more` is true.
- Proposed runtime batch defaults:
  - Query `page_size` = 50 (tunable).
  - Update pages sequentially or with small concurrency (3–5 workers) while respecting rate limits (≤2 req/s per token) and exponential backoff on 429.
- Reference patterns in repo for querying and filtering:
  - Example time-based filtering and query in `email_agent/notion_deals_client.py:74`–`email_agent/notion_deals_client.py:93` (uses created_time + filters).
  - External DB lookup by title in `research_agent/notion_updater.py:1432`–`research_agent/notion_updater.py:1461`.

## Document Retrieval (Per Page)
- Fetch full page content by traversing blocks recursively using `blocks.children.list` with pagination. Preserve headings, paragraphs, and list semantics when flattening to text.
- Partition content into individual report segments by any of:
  - Heading containing "🤖 Ava's Report".
  - Callout containing "Generated by Ava's Research Agent".
  - Sentinel markers like `=== REPORT START ===`.

## Extraction Pipeline (Hybrid)
1) Section-first parsing
- Within each report, locate a scoring section by headings such as "Scoring Assessment" or "Scoring Assessment with Confidence Indicators". If not found, widen to the whole report.

2) Regex/heuristics (no normalization stored)
- Match category names (case-insensitive) with tolerant punctuation/whitespace:
  - Founder Pattern
  - Idea Pattern
  - Structural Advantage
  - Asymmetric Signals or Asymmetric Signal (match both; store to property "Asymmetric Signal").
- Acceptable score formats beside the category mention:
  - Parenthetical fraction: `(17/25)`, `(24/35)`, `(3/5)`, `(7/10)` → store the numerator (e.g., `17`, `24`, `3`, `7`).
  - Inline fraction after separators: `: 17/25`, `– 7/10`, `= 3/5` → same handling as above.
  - Percent: `85%` → store `85` as the numeric value.
  - Plain number only if clearly scoped to the category and within a scoring context line.
- Explicitly skip unrelated 1–4 scales:
  - Ignore any `x/4`, `(x/4)`, or `x of 4` formats, even if found elsewhere on the page for other categories.
  - Note: For our four target categories, common denominators are 5 (Asymmetric), 10, 25, 35, and 100%.
- When multiple occurrences appear in a single report, choose the last occurrence inside the scoring section; otherwise last occurrence in the report.

3) LLM fallback (only if <3 categories found in a report)
- Use a low-cost model with a strict extract-only prompt:
  - JSON schema: `{ founder_pattern: {raw: string|null, matched: string|null}, idea_pattern: {...}, structural_advantage: {...}, asymmetric_signal: {...} }`
  - Instructions: "Return only scores that explicitly appear in the text. Do not infer or normalize. Echo the exact matched substring in `matched`."
- Post-validate by ensuring `matched` appears verbatim in the source text. Store `raw` exactly; discard any field without a verified `matched`.

## Multiple Reports Per Page (Pick Highest)
- Pages can contain several appended reports over time. For each category:
  - Compute a temporary comparable value for selection only (never stored):
    - Fraction `x/denom` → compare by `x/denom` if `denom > 0`.
    - Percent `p%` → compare by `p/100`.
    - Plain numbers: compare only against other plain numbers; ignore when mixed with fraction/percent.
  - Select the highest comparable candidate across all reports.
  - Store the numerator/explicit number in the Notion property; retain the raw string in the audit CSV for traceability.

## Updating Notion (Internal + External)
- Internal DB update:
  - Only patch properties that are empty or differ from the new best numeric value.
  - Update `number` properties with the extracted numerator/explicit score (e.g., `17` for `17/25`).
  - If extraction fails for a category, the script writes `-1` (and mirrors to the external DB when configured).
- External DB mirroring:
  - If `NOTION_DATABASE_ID_EXT` is set, find matching page by title property (`Name` → `Title` → `Project` → `Project Name`). See `research_agent/notion_updater.py:1443`–`research_agent/notion_updater.py:1461`.
  - Update the same four properties if they exist; do not change schema if missing.
  - External DB integration pattern mirrors existing code that already syncs status/priority/stage (`research_agent/notion_updater.py:1470`–`research_agent/notion_updater.py:1516`).

## Rate Limits, Resilience, and Idempotency
- Throttle to ~2 requests/second average; exponential backoff on HTTP 429.
- Pagination via `start_cursor` for both `databases.query` and `blocks.children.list`.
- `--dry-run` mode logs proposed changes without updating Notion.
- CSV audit log per run (written automatically to `scripts/notion_scores_backfill_audit.csv`): includes raw strings, numeric values, methods, and notes (e.g., `numeric_conversion_failed`).

## CLI Design (Separate Script; Planning Only)
- Path: `scripts/notion_scores_backfill.py`
- Arguments:
  - `--created-after 2025-06-01` (default enforced)
  - `--use-edited-time` (optional)
- `--page-size 50` with pagination via cursors
- `--max-items` (limit total processed pages; use `5` for initial test)
- `--dry-run` (no writes)
- `--max-workers 3` (concurrent page processors; default 3)
- `--no-llm` (optional override to disable GPT fallback; LLM enabled by default)
  - `--llm-model gpt-5-nano` (default; see cheap model notes)
- Env vars (loaded via `.env`):
  - `NOTION_API_KEY`
  - `NOTION_DATABASE_ID`
  - `NOTION_DATABASE_ID_EXT` (optional, for mirroring)
  - Provider-specific key for the LLM fallback (`OPENAI_API_KEY` for GPT-5-nano)

## Cheap LLM Fallback Options
- Note: Live pricing must be verified before deployment (cannot fetch here). Based on broadly available 2024–2025 info:
  - OpenAI `gpt-5-nano` — preferred default; very low cost while retaining solid extraction quality and already compatible with existing OpenAI integrations.
  - Google `gemini-1.5-flash` — low-cost, fast extraction; requires separate client setup.
  - Groq-hosted `Llama 3 8B` — very low cost/latency; needs stricter prompts and verification.
- Regardless of model, enforce extract-only JSON and substring verification.

## Pseudocode Outline
```
query = { filter: { created_time.after: 2025-06-01 }, page_size: N }
cursor = None; processed = 0
while True:
  resp = notion.databases.query(database_id, filter=query.filter, page_size=query.page_size, start_cursor=cursor)
  for page in resp.results:
    if processed >= max_items: stop
    text = fetch_blocks_recursive(page.id)
    reports = split_reports(text)
    candidates = [extract_scores(r) for r in reports]
    best = select_highest_per_category(candidates)  # temporary ratios only, never stored
    plan_updates = diff_with_existing(page.properties, best)
    if dry_run: log(plan_updates)
    else: update_internal(page.id, plan_updates); mirror_external_if_found(page.title, plan_updates)
    processed += 1
  if not resp.has_more: break
  cursor = resp.next_cursor
```

## Accuracy & Skip Rules
- Only accept scores explicitly present near the category label.
- Reject `x/4` formats entirely (used by unrelated categories elsewhere), even if found near our categories.
- Do not use overall score (e.g., `Score: 69/100`) as a category score.
- No normalization stored. Temporary normalization only for cross-report comparison.

## Observability & Outputs
- CSV audit log (default `scripts/notion_scores_backfill_audit.csv`) with raw strings, numeric values, extraction methods (`regex_structured`, `regex_global`, `llm_fallback`), and notes (e.g., `numeric_conversion_failed`).
- Summary: pages scanned, pages updated (internal/external), fields filled, fallback usage rate, skip counts.
- Concurrency logs surface parallel page processing; skipped pages (already filled) are tagged with `skipped_already_filled`.

## External Database Mirroring
- Enabled when `NOTION_DATABASE_ID_EXT` is set.
- Locate by title property using the same strategy as `NotionUpdater` (`research_agent/notion_updater.py:1443`–`research_agent/notion_updater.py:1461`).
- Update the four properties if present; do not attempt schema changes.

## Rollout
- Create properties in both databases.
- Dry-run 5 pages to validate behavior.
- Scale to batches of 50 with cautious rate limiting.
- Resume runs automatically skip pages whose four numeric fields are already populated (including `-1` failure markers).
- Monitor CSV and make targeted regex refinements if needed.

## Acceptance Criteria
- For pages created after 2025-06-01: all four properties populated with the highest valid raw strings across report segments when present.
- Unrelated 1–4 category scores are skipped.
- External DB receives mirrored updates when a matching page by title is found.
- Stored values capture the explicit numeric scores (e.g., numerator or percent value), while the audit log preserves the raw text for review.
